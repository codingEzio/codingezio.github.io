I was going for an explanation of what Netty is, why choose it specifically and when to use it that people could easily understand even without prior knowledge to the Java concurrency ecosystem (or as little as possible). Down below are my own understand of the whole thing, errors are expected.

And I think some of my confusions might be derived from inadequate understanding of web servers and only heard of Nginx and Apache, thought there were no alternatives.

Before We Begin

I do know a little bit of Django and Tornado before diving into the Java/Spring ecosystem. Both of them are web frameworks which have different capabilities, while they allow you writing code for things like the view template engine for user interaction, controller and models for the actual work, they also could serve the code by themselves which could be put directly onto the cloud (not recommended for production, but it could).

The name for the capability of serving the code are normally called as Application Server, which is just a normal web server (receive request -> handle -> return response) with much more capabilities (not just serving static resources, but could also executing code). They themselves "eliminate" the need for the core functionalities of actual web servers like Nginx, which now could serve as a mere proxy.

So, for Tornado, it's a web server (of course, you do need to write a few lines of code like the Express framework in Node.js) and a web framework.

Right now, it seems like that Netty serves like an Application Server while aims more than just web framework related stuff as it implemented protocols.

  • It is a NIO client server framework (quote)
  • It could do more than implementing an async application server
  • It handles things asynchronously which achieve faster speed (I/O)
  • It does the actual work and implementation for Transport Protocols
  • It was built on NIO (N: Non-Blocking) which is just I/O done asynchronously
