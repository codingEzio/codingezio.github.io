Kimi for Coding：403 伪装与 fallback

把 User-Agent 与 fallback 链一次配好，Kimi 才能稳定接入。

背景

接 Kimi for Coding 时，最容易撞上的不是模型本身，而是这句：

Kimi For Coding is currently only available for Coding Agents

根因并不复杂：Kimi for Coding 会检查 User-Agent。若请求头不像它认可的 Coding Agent，便直接回 403。

所以真正要说清的只有两件事：

  1. 如何伪装成可接受的 Coding Agent
  2. Kimi 不可用时，fallback 链如何接手

最短结论

若要让 Hermes 正常用上 Kimi for Coding，最短路径如下：

  1. base_url 指向 https://api.kimi.com/coding/v1
  2. 请求头带上可接受的 User-Agent
  3. 同时配好 fallback，别把单一供应商当成永不失手

已验证可工作的请求头之一：

default_headers:
  User-Agent: "claude-code/1.0"

配置范本

model:
  default: kimi-k2.5
  provider: kimi-coding
  base_url: https://api.kimi.com/coding/v1

default_headers:
  User-Agent: "claude-code/1.0"

fallback_providers:
  - provider: openrouter
    model: google/gemini-3-flash-preview
  - provider: openai-codex
    model: gpt-5.2-codex
  - provider: copilot
    model: gpt-5.2-codex

auxiliary:
  vision:
    provider: openrouter
    model: google/gemini-3-flash-preview

这里面真正关键者有三：

  • 主模型：kimi-k2.5
  • 入口地址：/coding/v1
  • 请求头：User-Agent: claude-code/1.0

少了第三项，最容易直接 403。

若源码里写死了旧 User-Agent

若项目不是纯配置驱动，而是把请求头硬写在代码里，就要把旧值一起改掉。

Before: KimiCLI/1.30.0
After:  claude-code/1.0

这一轮排查里，至少见过三类位置会写死它：

文件 作用
agent/auxiliary_client.py vision、压缩、flush 等辅助请求
run_agent.py 主 Agent 请求
hermes_cli/doctor.py 健康检查

fallback 该怎么排

这条链不求花，只求稳：

  1. Kimi：便宜，主打代码模型
  2. OpenRouter：拿来兜底，也顺手承接 vision
  3. OpenAI Codex
  4. GitHub Copilot

这样配置的好处是：

  • Kimi 可用时，优先吃最低成本主路
  • Kimi 被 403、429、402 或临时抽风时，不会整条链一起断
  • 非文本任务可提前绕到别家，少踩 Kimi 的限制面

验证点

改完之后，至少看三件事：

  1. Kimi 请求不再报 403
  2. 主链故障时，fallback 会继续出结果
  3. vision 请求没有继续误打到 Kimi 上

总结

此事本质很简单：

  • Kimi for Coding 不是随便一把 key 就能直连
  • User-Agent 过不了，便是 403
  • 伪装头配好后，还要把 fallback 一并配齐

至于“冷启动”那一段，这里不再作为公开结论保留；眼下真正稳定可复用的，还是 403 伪装 + fallback 配置 这两件事。
