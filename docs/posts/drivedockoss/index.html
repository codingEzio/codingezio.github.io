DriveDockOSS: 弹出磁盘之前，先看清楚是谁还在用它

一个为了“弹出前再确认一次”而做的原生 macOS 工具。

从一个私人工具开始

外接磁盘无法弹出时，macOS 往往只告诉你它“正在使用中”。真正需要回答的问题却更具体：是哪个 app、哪个进程、哪条路径，还是 Time Machine 仍在写入？

DriveDock 最初只是我给自己做的小工具。它逐渐能显示磁盘容量、设备信息、打开的文件句柄和 Time Machine 状态，也能在确认后请求 macOS 正常弹出磁盘。功能稳定之后，我决定保留原来的私人项目，同时整理出独立的开源版本：DriveDockOSS。

它具体做什么

DriveDockOSS 同时提供 menu bar popover 和可调整大小的桌面窗口。一次扫描会汇总：

  • 已挂载的内置与外接磁盘；
  • 可用容量、设备型号、连接协议与 SMART 状态；
  • lsof 可见的进程、PID、访问模式与路径；
  • Time Machine destination 与当前备份状态；
  • 单盘与批量弹出的风险提示。

它也可以向普通 GUI app 发送 macOS 的正常退出请求，但不会 force quit。VM 与 container host 有自己的关机语义，因此仍然要求用户回到对应 app 手动处理。

安全模型比“更强硬”重要

DriveDockOSS 的目标不是绕过 macOS，而是把弹出之前的信息补全。

扫描使用只读系统命令。内置磁盘始终只监控；弹出必须经过确认，并且最终只调用正常的 diskutil eject。如果 Time Machine 正在使用目标磁盘，app 会先询问是否停止备份、等待它进入 idle，再请求 macOS 弹出。

这里没有 force-eject 路径。lsof 在普通用户权限下也只能提供 best-effort 结果，所以 UI 会明确说明：没有看到句柄，不等于系统保证没有活动。

为开源重新整理

这次不是把私人 repo 原样公开。我创建了独立的 DriveDockOSS 项目，并重新检查了文件边界与 Git 历史。

公开 repo 使用 default-deny .gitignore：源文件、测试、默认配置、README、license、contribution guide、icon 和经过检查的截图需要被明确允许。agent 指令、本地签名配置、build output、日志、用户路径和机器状态不会进入公开历史。

公开历史从一个干净的 root commit 开始，作者使用 GitHub noreply 地址。项目也带有 tracked commit-msg hook；贡献者运行 make setup-hooks 后，本地 Git 与 GitHub Desktop 都会检查 Conventional Commits 格式。

UI 不应该要求猜缩写

早期界面为了节省空间使用过 H、R、W、P 一类指标缩写。它们对作者也许熟悉，对第一次使用的人却意味着额外猜测和 hover。

现在对应指标直接显示为 Handles、Read、Write 和 Processes。popover 的宽度、高度比例与上下限也都进入配置，不再只适合某一台机器的窗口尺寸。

设置页还支持 English、简体中文、繁体中文、韩语与日语。语言和 tone 分开建模，因此标准文案之外也可以选择 Gen Z tone，而不需要复制整套产品逻辑。

macOS 兼容与发布现实

当前 release 面向 Apple silicon，deployment target 是 macOS 15.6，并已在 macOS 15.6.1 VM 中运行验证。macOS 26 可以使用 Liquid Glass；较旧系统使用原生 material fallback。

我目前没有 Apple Developer Program membership，因此 GitHub Release 中的 app 是 ad-hoc signed、未 notarize 的 build。macOS 可能要求用户在第一次运行时手动批准。源码与 build scripts 都在 repo 中，也可以按照 README 本地构建。

首个版本是 v0.1.0。

下一步

接下来会继续观察真实磁盘、Time Machine 与不同外接盒场景，补充更适合公开展示的假数据截图，并在条件允许时加入 Developer ID signing 与 notarization。

这次开源真正想保留的不是“一个能弹盘的按钮”，而是一条简单边界：先让风险变得可见，再让用户决定是否继续。
